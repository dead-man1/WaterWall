# PacketToConnection Node

`PacketToConnection` is a layer-3 bridge that feeds IPv4 packets into lwIP and turns the resulting transport activity into Waterwall lines.

In other words:

- upstream packet payload is treated as incoming IP traffic
- TCP activity inside lwIP can create normal Waterwall connection lines
- payload from those connection lines is written back into lwIP
- packets generated by lwIP are emitted downstream as raw IP packets

This tunnel is how Waterwall can translate between packet-oriented chains and connection-oriented chains.

## What It Does

- Accepts raw IPv4 packets on the upstream payload path.
- Creates lwIP virtual interfaces on demand per destination IP.
- Opens TCP listeners on demand when a new destination TCP port appears.
- Creates a Waterwall line when lwIP accepts a TCP connection.
- Forwards TCP payload from lwIP into the next tunnel as normal upstream payload.
- Writes downstream payload from the next tunnel back into the lwIP TCP connection.
- Emits packets generated by lwIP back toward the previous tunnel.

## Typical Placement

A common layout is:

- some packet-producing node
- `PacketToConnection`
- some connection-oriented service chain

Examples of packet-producing nodes include:

- `TunDevice`
- `WireGuardDevice`
- `RawSocket`

Example conceptual flow:

- packet source -> `PacketToConnection` -> HTTP server chain
- responses from the HTTP server chain -> `PacketToConnection` -> raw packets back to packet source

## Configuration Example

```json
{
  "name": "packet-to-connection",
  "type": "PacketToConnection",
  "next": "service-side-node"
}
```

## Required JSON Fields

### Top-level fields

- `name` `(string)`
  A user-chosen name for this node.

- `type` `(string)`
  Must be exactly `"PacketToConnection"`.

### `settings`

There are currently no tunnel-specific JSON settings.

Tunnel creation initializes the lwIP stack directly and does not parse a settings object for runtime behavior.

## Detailed Behavior

### Packet ingress model

Upstream payload is expected to be a full IP packet.

Current implementation behavior:

- IPv4 packets are accepted
- IPv6 packets are dropped
- only TCP and UDP protocol numbers are recognized
- unsupported protocols such as ICMP are ignored or dropped

For each destination IPv4 address seen in the packet stream, the tunnel creates a per-address lwIP `netif` on demand.

### TCP path

When a TCP SYN arrives for a destination port that has not been seen before:

- the tunnel creates a listening lwIP TCP PCB bound to that destination IP and destination port
- later, when lwIP accepts a connection, a real Waterwall `line_t` is created
- the line routing context is populated from the TCP tuple
- upstream `Init` is sent into the next tunnel

After that:

- TCP payload received by lwIP is copied into `sbuf_t` buffers and forwarded as upstream payload
- remote TCP close or error triggers upstream `Finish`
- downstream payload from the next tunnel is written into lwIP with `tcp_write()`

This is the fully implemented path in the current codebase.

### Packet egress model

Packets generated by lwIP are emitted through the tunnel's virtual `netif` output callback.

The callback:

- copies the outgoing pbuf into an `sbuf_t`
- sends that packet downstream toward the previous packet-oriented side

So the previous tunnel sees raw packets again, not connection payload.

### Backpressure and pause/resume behavior

`PacketToConnection` participates in Waterwall flow control for TCP:

- if `tcp_sndbuf()` cannot accept the whole downstream payload, the remainder is queued
- upstream is paused with `tunnelNextUpStreamPause()`
- queued data is flushed later from the lwIP sent callback
- once the queue drains, upstream is resumed with `tunnelNextUpStreamResume()`

The reverse direction also uses pause/resume:

- downstream `Pause` stops acknowledging received TCP bytes back to lwIP
- downstream `Resume` calls `tcp_recved()` for the withheld amount

This lets the connection side apply backpressure without breaking Waterwall's direction ownership rules.

### Internal line lifetime model

Per-line state keeps:

- a reference to the owning tunnel
- the Waterwall line
- the lwIP PCB
- queued write buffers
- queued acknowledgements

The line is manually locked once during line-state initialization and only fully released when all pending worker messages are done and the line state is destroyed.

That matches the cross-thread callback model used by lwIP worker messages in this tunnel.

### UDP status in the current implementation

The upstream packet path does create and bind UDP PCBs on demand.

However, the current UDP receive callback only frees incoming packets:

- it does not create Waterwall lines
- it does not forward UDP payload to the next tunnel

So UDP support is only partial right now.

## Notes And Caveats

- Current implementation is effectively TCP-focused. That is the complete and usable path today.
- IPv6 is not supported in this tunnel at the moment.
- Upstream `Init` is not normally called from outside; lines are created internally when lwIP accepts TCP connections.
- Downstream `Init` is marked impossible in the current code and aborts if called.
- The tunnel has no JSON settings today, so configuration is mostly about where you place it in the chain.
