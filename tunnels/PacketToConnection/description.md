# PacketToConnection Node

`PacketToConnection` is a layer-3 to layer-4 bridge that feeds IPv4 packets into lwIP and turns resulting transport activity into Waterwall connection lines.

In other words:

- upstream packet payload is treated as incoming IP traffic
- TCP activity inside lwIP can create normal Waterwall connection lines
- payload from those connection lines is written back into the lwIP connection
- packets generated by lwIP are emitted downstream as raw IP packets

## TunMode Client-Side Intent

`PacketToConnection` is designed for client-side **TunMode** use cases where the application already understands destination packet semantics and reconstructs internal connection behavior.

That allows the core to receive meaningful connection context (source/destination tuple) and then apply internal routing logic (for example routing decisions, domain-based handling pipelines, and policy chains).

## PacketToConnection vs PacketsToStream

- `PacketsToStream` is a framing adapter. It just converts packet boundaries to/from stream framing (`2-byte size + payload`) and does not terminate transport sessions.
- `PacketToConnection` is a protocol bridge. It injects packets into lwIP, terminates/creates transport sessions, and exposes per-connection Waterwall lines.

Choose `PacketsToStream` when you only need packet boundary preservation over a stream transport.

Choose `PacketToConnection` when you need connection-aware handling from packet input (especially TunMode client-side logic).

## What It Does

- Accepts raw IPv4 packets on the upstream payload path.
- Creates lwIP virtual interfaces on demand per destination IP.
- Opens TCP listeners on demand when a new destination TCP port appears.
- Creates a Waterwall line when lwIP accepts a TCP connection.
- Forwards TCP payload from lwIP into the next tunnel as normal upstream payload.
- Writes downstream payload from the next tunnel back into the lwIP TCP connection.
- Emits packets generated by lwIP back toward the previous tunnel.

## Typical Placement

A common layout is:

- packet-producing node (for example `TunDevice`, `WireGuardDevice`, or `RawSocket`)
- `PacketToConnection`
- connection-oriented service chain

Example conceptual flow:

- packet source -> `PacketToConnection` -> service chain
- service chain responses -> `PacketToConnection` -> raw packets back to packet source

## Configuration Example

```json
{
  "name": "packet-to-connection",
  "type": "PacketToConnection",
  "next": "service-side-node"
}
```

## Required JSON Fields

### Top-level fields

- `name` `(string)`
  A user-chosen name for this node.

- `type` `(string)`
  Must be exactly `"PacketToConnection"`.

### `settings`

There are currently no tunnel-specific JSON settings.

Tunnel creation initializes lwIP directly and does not parse a settings object for runtime behavior.

## Detailed Behavior

### Packet ingress model

Upstream payload is expected to be a full IP packet.

Current implementation behavior:

- IPv4 packets are accepted
- IPv6 packets are dropped
- only TCP and UDP protocol numbers are recognized
- unsupported protocols (for example ICMP) are ignored/dropped

For each destination IPv4 address seen in the packet stream, the tunnel creates a per-address lwIP `netif` on demand.

### TCP path

When a TCP SYN arrives for a destination port that has not been seen before:

- the tunnel creates a listening lwIP TCP PCB bound to that destination IP and destination port
- later, when lwIP accepts a connection, a real Waterwall `line_t` is created
- line routing context is populated from the TCP tuple
- upstream `Init` is sent into the next tunnel

After that:

- TCP payload received by lwIP is forwarded as upstream payload on that line
- downstream payload from the next tunnel is written into lwIP with `tcp_write()`
- remote close/error triggers upstream `Finish`

This is the complete and usable path in the current codebase.

### Packet egress model

Packets generated by lwIP are emitted through the tunnel's virtual `netif` output callback.

The callback copies outgoing pbuf content into an `sbuf_t` and sends it downstream toward the previous packet-oriented side.

### Backpressure and pause/resume behavior

`PacketToConnection` participates in Waterwall flow control for TCP:

- if `tcp_sndbuf()` cannot accept full downstream payload, remainder is queued
- upstream is paused with `tunnelNextUpStreamPause()`
- queued data is flushed later from lwIP sent callbacks
- once queue drains, upstream is resumed with `tunnelNextUpStreamResume()`

The reverse direction also uses pause/resume:

- downstream `Pause` defers `tcp_recved()` acknowledgements
- downstream `Resume` releases deferred acknowledgements

### Internal line lifetime model

Per-line state keeps:

- tunnel reference
- Waterwall line
- lwIP PCB
- queued write buffers
- queued acknowledgements

The line is kept alive across cross-thread callbacks and is fully released only after pending worker messages are done and line state is destroyed.

### UDP status in current implementation

The upstream packet path does create and bind UDP PCBs on demand.

However, current UDP receive callback only frees packets and does not create Waterwall lines or forward UDP payload to the next tunnel.

So UDP support is partial right now.

## Notes And Caveats

- Current implementation is effectively TCP-focused.
- IPv6 is not supported in this tunnel at the moment.
- Upstream `Init` is not normally called from outside; lines are created internally when lwIP accepts TCP connections.
- Downstream `Init` is marked impossible in current code and aborts if called.
- The tunnel has no JSON settings today, so behavior is mainly defined by placement in the chain.
