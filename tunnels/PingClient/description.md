# PingClient Node

`PingClient` is a layer-3 packet tunnel that wraps outgoing IPv4 packets into real IPv4 ICMP echo-request packets and unwraps the matching echo-request tunnel packets on the return path.

It is a pure packet tunnel created with `packettunnelCreate()`, so it does not create per-connection line state and it works on the worker packet lines supplied by the chain.

## What It Does

- upstream packet payload is encapsulated as:
  outer IPv4 header -> ICMP echo header -> original IPv4 packet
- downstream packet payload is decapsulated when it matches the configured tunnel endpoint IPs and ICMP identifier
- outgoing tunnel packets are always ICMP echo requests
- no ICMP echo replies are generated by this tunnel
- `xor-byte` can XOR the ICMP payload bytes before transmission and restore them on decapsulation
- `roundup-size` can add a 2-byte original-size prefix and pad the ICMP payload up to `56`, `128`, `256`, `512`, `1024`, or the maximum supported ICMP payload size
- the original inner IPv4 packet checksum is refreshed before encapsulation if the line requested checksum recalculation
- the freshly created outer IPv4 and ICMP checksums are calculated immediately
- total packet size is capped at `1500` bytes, so oversized payloads are logged and dropped

## Required `settings`

- `local-ip` `(string)`
  Outer IPv4 source address used when this client encapsulates upstream traffic.

- `peer-ip` `(string)`
  Outer IPv4 destination address used when this client encapsulates upstream traffic.

## Optional `settings`

- `identifier` `(integer)`
  ICMP echo identifier.
  Default: `44975` (`0xAFAF`)

- `sequence-start` `(integer)`
  Initial ICMP echo sequence number counter.
  Default: `0`

- `ipv4-id-start` `(integer)`
  Initial outer IPv4 identification counter.
  Default: `0`

- `ttl` `(integer)`
  Outer IPv4 TTL.
  Default: `64`

- `tos` `(integer)`
  Outer IPv4 TOS byte.
  Default: `0`

- `xor-byte` `(integer)`
  XOR byte applied to the ICMP payload only.
  Omit this field to disable XOR.

- `roundup-size` `(boolean)`
  When `true`, the ICMP payload stores:
  `2-byte original-size -> original IPv4 packet -> random filler`
  and rounds the payload length up to the next supported bucket.
  Default: `false`

## Example

```json
{
  "name": "icmp-client",
  "type": "PingClient",
  "settings": {
    "local-ip": "198.51.100.10",
    "peer-ip": "203.0.113.20",
    "identifier": 4660,
    "xor-byte": 90,
    "roundup-size": true,
    "sequence-start": 0,
    "ttl": 64
  },
  "next": "raw-out"
}
```

## Notes

- only IPv4 inner packets are encapsulated by this tunnel
- only matching IPv4 ICMP echo-request envelopes are decapsulated on the return path
- fragmented outer ICMP packets are not reassembled here, so they are passed through unchanged
- with `roundup-size`, packets larger than `1470` bytes are dropped because the 2-byte size prefix must still fit inside the `1500` byte total packet limit
